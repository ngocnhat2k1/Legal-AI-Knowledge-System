# Customs Assistant

An internal tool for the customs-declaration staff of a Vietnamese logistics company.

It answers two questions a declarant asks every day:

1. **What is the duty on this shipment?** — looked up by exact key `(HS code, schedule, date)`, with
   the governing decree, the data snapshot date, and the conditions attached.
2. **Which HS code is this good?** — answered with **three candidates and the verbatim chapter and
   section notes that support each**, for a human to decide. Never a bare code.

## The Two Rules That Govern Everything

**Tariff rates are never generated by an LLM.** They are looked up by exact key in SQL. Semantic
search over a tariff table returns the most similar-looking row, and in a tariff table the
most similar-looking row is usually the wrong row — the nomenclature is built out of near-identical
sibling lines that differ by one qualifier and by tens of percentage points.

**HS codes are suggested, never asserted.** Independent benchmarks put autonomous LLM accuracy at
29–47% at 10-digit against a 95% human baseline. The same models reach 93.9% when they return three
candidates with retrieved evidence and let a human decide. The gap is the output contract, not model
capability.

Both exist because the failure mode here is silent. A wrong HS code is a real, correctly-formatted
code — no exception, no parse error, no red flag. It clears customs and surfaces years later as a
post-clearance audit with back-assessment, penalties, and daily interest. The declarant carries that
liability, not the tool.

Consequently: this is a **research aid that shows its sources and refuses when its data may be
stale**, never an answer engine that states a rate.

## Why There Is So Much Documentation And So Little Code

Vietnamese customs law is more volatile than software. During research, one decree was signed and
became legally binding **the same day**, was published in machine-readable form **fifteen days
later**, and **expired fifty-two days after that** with rates silently reverting. Half the laws named
in the original project roadmap were already dead — one had been replaced seventeen days earlier.

None of that is inferable from code, and no model's training data has it. The
[`.agent/concepts/`](.agent/concepts/) notes carry it, with a verification date and source on every
factual claim, and an explicit `Unverified / Do Not Rely On` section on every note. Those sections are
load-bearing — do not tidy them into confident prose.

## Provenance

The knowledge base was built on 2026-07-17 from twelve research agents that fetched live sources,
including three adversarial verification passes. Where two reports contradicted each other, both are
recorded and the conflict is marked unresolved rather than silently settled.

The notes were then audited against their own sources. The audit caught the writing agents grafting a
real URL onto an unsourced claim, inventing a provenance phrase, and asserting a decree union that
appears in no report — the same laundering-of-uncertainty this product exists to prevent, committed by
the tooling that documented it. All fixed. **Re-audit whenever the knowledge base is materially
extended:** fluent prose hides missing provenance.

The scaffold came from [APB](https://github.com/truongthuc/apb), which is maintained in its own
repository and is not vendored here.
